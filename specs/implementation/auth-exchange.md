# ZyncBase Authentication Exchange

## 1. Core Principle

ZyncBase is a **resource server**, not an identity provider. It does not manage accounts, passwords, OAuth flows, memberships, billing state, or permission truth.

ZyncBase accepts identity from two configured sources:

1. **External JWTs** issued by the application identity provider.
2. **Anonymous subjects** generated by the SDK when anonymous authentication is enabled.

The server validates the presented identity material, projects trusted claims into `$session`, issues a short-lived connection ticket, and enforces `authorization.json` natively in Zig.

## 2. The `$session` Context

`$session` is the unified source of truth for ZyncBase-side authorization. Rules never inspect raw JWTs directly.

- **JWT source**: Configured claim projection from a validated external JWT.
- **Anonymous source**: SDK-generated high-entropy subject with `isAnonymous = true`.
- **Identity**: `$session.userId` is always the internal `users.id` resolved through SQLite, never a raw JWT subject or anonymous subject.
- **Permissions**: Permission data must already exist in `$session` claims, namespace parts, the target row, or the incoming value. ZyncBase does not load user-row fields or perform relationship lookups for authorization.
- **Persistence**: Fixed for a resolved scope until auth refresh or namespace switching requires re-resolution.

Example JWT-backed session:

```json
{
  "sub": "auth0|123",
  "tenant_id": "acme",
  "role": "admin",
  "read_projects": ["docs", "wiki"],
  "write_projects": ["docs"]
}
```

Example anonymous session:

```json
{
  "sub": "anon:6c6f8b0d0f1c4d4c9d61b7fdc4f80f52",
  "isAnonymous": true,
  "role": "anonymous"
}
```

## 3. The Connection Lifecycle

ZyncBase uses a mandatory ticket-based handshake so JWTs are never placed in WebSocket URLs.

### Step 1: Ticket Request

For authenticated users, the SDK sends the external JWT to the ZyncBase HTTP endpoint:

```http
POST /auth/ticket
Authorization: Bearer <external_jwt>
```

For anonymous users, the SDK sends its persisted anonymous subject when anonymous authentication is enabled:

```http
POST /auth/ticket
Content-Type: application/json

{ "anonymousSubject": "anon:6c6f8b0d0f1c4d4c9d61b7fdc4f80f52" }
```

The anonymous subject is a bearer credential. The SDK must generate it from high-entropy randomness and persist it locally.

### Step 2: Identity Validation and Session Projection

For JWT authentication, Zig validates:

1. signature
2. issuer
3. audience
4. expiration and not-before times
5. configured algorithm and key constraints

ZyncBase then projects configured claims into the base `$session`.

For anonymous authentication, Zig validates that anonymous auth is enabled, validates the anonymous subject format and entropy requirements, and creates the base anonymous `$session`.

No user-row fields are loaded during this step.

### Step 3: Ticket Issuance

Zig generates a short-lived, single-use ticket. The ticket is a signed opaque blob containing the external subject, base `$session`, issue time, expiry, and replay-prevention identifier.

```json
{
  "ticket": "zyc_tk_8s2k...",
  "expiresAt": 1741551234
}
```

### Step 4: WebSocket Connection

The SDK connects to the WebSocket with the ticket in the query string:

```text
ws://server/ws?ticket=zyc_tk_8s2k...
```

Zig verifies the ticket signature, expiry, and single-use state, then hydrates transport state from the base `$session`.

Store and presence operations are still gated until their namespaces are selected and scoped `users.id` values are resolved.

### Step 5: Scoped Readiness

The SDK sends the initial store and presence namespace selections as part of its connection flow. For each scope, Zig resolves:

1. the namespace string to `_zync_namespaces.id`
2. the external subject to `users.id`
3. `$session.userId` for that scope

`connect()` resolves only after required initial scopes are ready.

If `users.namespaced = false`, identity resolution uses reserved global namespace ID `0`. If `users.namespaced = true`, identity resolution uses the scope namespace ID.

## 4. Token Refresh

External JWTs and tickets are short-lived, while WebSockets may be long-lived. The SDK can rotate the session without disconnecting.

1. **Client**: Sends `AuthRefresh` over the existing WebSocket with a new external JWT.
2. **Zig**: Re-validates the new JWT (signature, expiry, issuer, audience).
3. **Zig**: Projects a new `$session` from the validated JWT claims and updates the connection's token expiry.
4. **Zig**: Sends an `ok` response with the updated `$session`. Active scopes continue without interruption.

If validation fails, the server sends `ServerDisconnect` with code `AUTH_FAILED` and closes the connection.

Subsequent operations on the same connection evaluate against the updated scoped `$session`.

Short-lived tokens plus refresh are the normal permission-update and revocation path. ZyncBase does not maintain a server-side permission database or revocation list by default.

## 5. Security & Performance

- **Zero-Trust Handshake**: Zig never accepts a WebSocket connection without a valid ticket.
- **No JWTs in URLs**: JWTs are exchanged over HTTP headers or refresh frames, not WebSocket query strings.
- **Native Authorization**: `$session` is baked into scoped connection state, so `authorization.json` evaluation happens natively in Zig with no foreign calls.
- **No Hidden Auth Reads**: The `users` table is resolved for internal identity and ownership references only. Profile fields are application data, not authorization inputs.
- **Bounded Permissions**: Permission claims should remain human-manageable. Applications should use roles, groups, scoped grants, or active-context tokens instead of unbounded permission lists.

## 6. Error Conditions

| Step | Condition | HTTP/WS Response |
|------|-----------|-----------------|
| Ticket Request | JWT signature invalid or expired | HTTP 401 `AUTH_FAILED` |
| Ticket Request | JWT issuer, audience, or algorithm invalid | HTTP 401 `AUTH_FAILED` |
| Ticket Request | Anonymous auth disabled | HTTP 401 `AUTH_FAILED` |
| Ticket Request | Anonymous subject invalid | HTTP 400 `INVALID_MESSAGE` |
| WebSocket Upgrade | Ticket expired | HTTP 401 `AUTH_FAILED` |
| WebSocket Upgrade | Ticket already used | HTTP 401 `AUTH_FAILED` |
| WebSocket Upgrade | Origin not in `allowedOrigins` | HTTP 403 |
| `AuthRefresh` | New JWT invalid or expired | `ServerDisconnect` `AUTH_FAILED`, connection closed |
| `AuthRefresh` | Subject mismatch (different user) | `ServerDisconnect` `AUTH_FAILED`, connection closed |

In all failure cases, the connection is not established, or the connection is terminated if already connected. No partial session state is retained.
