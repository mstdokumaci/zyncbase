# Unify the operator model — collapse `types.ComparisonOp` into `query_ast.Operator`

## Honest assessment first

Your original three-helper idea (`isComparison`/`needsValue`/`isTextMatch`) **does not fit the code**:
- `isComparison = {eq,ne,gt,lt,gte,lte}` is split in 1 of 3 sites (`query_parser.zig:353` allows arrays only for `{eq,ne}`; `{gt,...}` reject them).
- `isTextMatch = {contains,startsWith,endsWith}` is **wrong**: `contains` has array-contains semantics that `startsWith`/`endsWith` lack; the real recurring pair is `{startsWith,endsWith}`.

But the underlying observation is real: the "13-arm switch is in 6+ places" pain comes from **two parallel operator enums** kept in sync by a 1:1 bridge. That's the actual duplication. You chose unification, which is the only option that kills the bridge outright. Plan below.

## Current state (verified)

Two distinct enums:
- **`query_ast.Operator`** (`query_ast.zig:7`) — 13 arms: `eq, ne, gt, lt, gte, lte, contains, startsWith, endsWith, in, notIn, isNull, isNotNull`. Wire: numeric `u8` via `@enumFromInt` (`query_parser.zig:238`).
- **`types.ComparisonOp`** (`authorization/types.zig:133`) — 9 arms: `eq, ne, gt, gte, lt, lte, in_set, not_in_set, contains`. Wire: string keys `"in"`/`"notIn"` from JSON (`parse.zig:236`).

Bridge: `mapToQueryOp()` (`doc_predicate.zig:452`) is a **pure 1:1 identity** — `in_set→in`, `not_in_set→notIn`, everything else identity. This is the signature of accidental duplication.

`ComparisonOp` is **purely authorization-internal**: re-exported at `authorization.zig:16` but consumed by nothing outside `src/authorization/` (verified — zero non-test references elsewhere). No import cycle: `query_ast.zig` imports only `std`/`schema`/`typed`.

Migration surface: **11 non-test `in_set`/`not_in_set` sites** across 4 files + 1 re-export + ~5 references in `authorization_test.zig`.

## The plan

### Step 1 — Canonical home: `query_ast.Operator` (unchanged, 13 arms)
No enum changes. `query_ast.zig` is the right home — auth already depends on query_ast one-way. Auth is a *source* of predicates; `query_ast` is the target query model.

### Step 2 — Rename auth enum arms: `in_set`→`in`, `not_in_set`→`notIn`
This is the crux. It makes `ComparisonOp` a literal subset of `query_ast.Operator`, so `mapToQueryOp` becomes identity.

Safe because:
- `.in`/`.notIn` are **not Zig keywords** — the query enum already uses them without issue.
- **Wire formats are untouched**: auth config JSON keeps string keys `"in"`/`"notIn"` (already documented at `specs_llm/.../auth-system.txt:54`); query wire keeps numeric codes. The `_set` suffix was purely a Zig-land name.

Sites to rename (4 files, 11 references):
- `authorization/types.zig:140-141` — enum declarations
- `authorization/parse.zig:236-237` — StaticStringMap values (`.{ "in", .in }`, `.{ "notIn", .notIn }`)
- `authorization/evaluate.zig:387,391` — `compareValues` switch arms
- `authorization/doc_predicate.zig:333,397,448,460,461` — validate + capability + bridge

### Step 3 — Delete the bridge and the duplicate enum
- **Delete `mapToQueryOp()`** (`doc_predicate.zig:452-464`). At its sole call site (`doc_predicate.zig:226`), replace `.op = mapToQueryOp(comp.op)` with `.op = comp.op` — now a direct assignment since the types unify.
- **Delete `types.ComparisonOp`** (`authorization/types.zig:133-143`).
- **Delete the `authorization.zig:16` re-export** (`pub const ComparisonOp = types.ComparisonOp;`).
- In `authorization/types.zig:124`, change `Comparison.op: ComparisonOp` → `op: query_ast.Operator` (requires adding `const query_ast = @import("../query_ast.zig");` to types.zig).
- Propagate the type change through `doc_predicate.zig` signatures (`validateLiteralValue:328`, `validateContextVarValue:360`, `validateRhsType:393`, `operatorAllowedForField:443` — all `op: types.ComparisonOp` → `op: query_ast.Operator`) and `evaluate.zig:373` (`compareValues`).

### Step 4 — Handle the 4 query-only arms in auth
Auth switches over the auth enum currently have 9 arms and are exhaustive. After unification they switch over the 13-arm `query_ast.Operator` and **must** become exhaustive over all 13. Since the auth parser (`parseComparisonOp`) only ever produces the 9 documented ops, the 4 extra arms (`startsWith`, `endsWith`, `isNull`, `isNotNull`) can never occur — but Zig's exhaustive-switch check doesn't know that.

Two switches need completing:
- **`operatorAllowedForField`** (`doc_predicate.zig:443`): add the 4 arms returning `false` (they're unsupported in auth predicates). This is **not** extending auth semantics — it's explicitly marking them unsupported. Future work to add them is a separate feature.
- **`compareValues`** (`evaluate.zig:373`): add the 4 arms returning `false` (unreachable in practice but required for exhaustiveness), OR — cleaner — add an `else => unreachable` arm with a comment, since parse guarantees they can't arrive here. **Decision point for you** (see question below).

The other auth switches (`validateLiteralValue:332`, `validateRhsType:392`) already use `else =>` fallback arms and need no change.

### Step 5 — Update `authorization_test.zig`
~5 references to `.in_set`/`.not_in_set` (lines 266, 669, 677, 1006, 1024, 1044) → rename to `.in`/`.notIn`. The type annotation (if any) `types.ComparisonOp` → `query_ast.Operator`. The existing `expectEqual(query_ast.Operator.eq, condition.op)` assertions at lines 470/719/720 stay as-is — they already assert against `query_ast.Operator`, which confirms the bridge was pure identity.

### Step 6 — No spec changes required
`specs_llm/.../auth-system.txt:54` already documents operators as `in`/`notIn`. The internal rename aligns naming with the already-documented wire format. No `npm run specs:compress` needed (specs unchanged).

## What this plan does NOT do
- Does **not** extend auth semantics to support `startsWith`/`isNull`/etc. — those remain query-only. That's a separate feature request.
- Does **not** change the original three-helper proposal's `isComparison`/`needsValue`/`isTextMatch`. Those helpers are still not worth adding (per the assessment above). If after unification you want a small `comptime` capability helper shared by both sides, that's a natural follow-up — but unification alone removes the pressure.
